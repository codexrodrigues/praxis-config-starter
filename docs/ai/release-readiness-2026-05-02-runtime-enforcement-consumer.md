# Release Readiness - Runtime Enforcement Consumer Checkpoint

Date: 2026-05-02

Status: source-level beta checkpoint, locally proven with Neon-backed
quickstart and local browser proof.

## Scope

This report records the local-first readiness checkpoint for runtime consumers
of governed semantic decisions authored and published through `domain-rules`.

The validated path covers:

- `form_config` materialization consumed by the official Angular dynamic-form
  runtime;
- `backend_validation` materialization consumed by a mutable backend command;
- `workflow_action` materialization consumed by an existing operational action;
- `approval_policy` materialization consumed by an existing bulk operational
  action.

This checkpoint does not authorize Maven Central publication, npm publication,
hosted smoke execution or broadening the runtime materialization contract.

## Contract State

The current beta contract is intentionally narrow:

- `/api/praxis/config/domain-rules/**` remains the source of truth.
- Runtime hosts consume applied materializations; they do not author or redefine
  the semantic decision.
- `form_config` is a derived UI/runtime projection.
- `backend_validation` uses `targetArtifactType=resource-validation` and
  `kind=resource_validation_policy`.
- `workflow_action` uses `targetArtifactType=resource-workflow-action` and
  `kind=workflow_action_policy`.
- `approval_policy` uses `targetArtifactType=resource-action-approval` and
  `kind=approval_policy`.
- Applied runtime materializations must include a derived `sourceHash` when the
  projection is generated by publication.
- Page Builder, Angular components, FormConfig, controllers and services remain
  consumers or cockpit surfaces, not primary rule sources.

This keeps Praxis aligned with the platform premise: AI authors governed
semantic decisions, and runtimes consume derived, auditable projections.

## Bug Avoided During Proof

A stale quickstart process on `http://localhost:8088` still embedded
`praxis-config-starter-0.1.0-rc.5` while the repository `pom.xml` pointed to
`0.1.0-rc.37`.

That stale process produced false `500` failures during procurement probes.
Before diagnosing runtime enforcement drift, verify the active jar and embedded
starter version.

## Release State

Ready as an internal source-level beta checkpoint for runtime enforcement
consumers of governed semantic decisions.

Not ready for external artifact publication by itself.

Do not create a new Maven Central or npm version solely for this checkpoint
unless all of these become true:

- a named downstream consumer needs the runtime enforcement proof from public
  artifacts;
- the owner explicitly authorizes a named release cut;
- quickstart is freshly packaged with the intended starter version;
- dynamic-form browser proof passes freshly;
- backend-validation, workflow-action and approval-policy smokes pass freshly
  against Neon;
- one intentionally selected remote release gate is accepted.

## Residual Risks

- The proof is local-first and Neon-backed, not a hosted published smoke.
- The existing all-in-one `verify-domain-rules-runtime.sh` still carries broader
  option-source and timeline paths; focused lanes should be preferred for
  targeted diagnostics.
- The cockpit/docs still need a coherent user-facing presentation of all three
  non-UI projections before this should become a public product claim.
- Repeating the smokes writes new persisted definitions/publications into Neon
  under isolated tenant ids; this is acceptable for gates but should not be run
  casually in tight loops.

## Next Recommendation

Do not open a new materialization family immediately.

First consolidate cockpit and public documentation so operators can understand
the proven projections:

- visual/runtime guidance via `form_config`;
- command blocking via `backend_validation`;
- action blocking via `workflow_action`;
- approval requirement via `approval_policy`.
